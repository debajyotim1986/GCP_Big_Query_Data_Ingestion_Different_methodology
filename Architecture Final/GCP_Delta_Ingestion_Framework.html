📌 Problem Statement

What This Framework Must Do

1

Detect delta records between Raw & Staging layer (same schema, both in BigQuery)

2

Ingest delta records into Staging table as a new partition

3

Flag the new partition as UNPROCESSED in a metadata table

4

Pick up unprocessed partitions and apply transformation logic

5

Load transformed data into the Curation layer

Delta Detection Strategies

These fundamental strategies are used across all 6 architectures. Choose based on your data volume, schema stability, and latency requirements.

🔎
EXCEPT ALL
SELECT * FROM raw EXCEPT ALL SELECT * FROM staging
#️⃣
SHA256 Hash
TO_HEX(SHA256( CONCAT(col1,col2,...) ))
🔗
PK + Hash Join
LEFT ANTI JOIN on PK + hash mismatch check
⏱️
Watermark
WHERE updated_at > MAX(staging.updated_at)
📊
Row Count + Checksum
COUNT(*) + SUM(hash) per partition

Secret Manager
Design Intent: Cloud Composer is scheduled on the last day of every month. On trigger, it spins up a fresh ephemeral Dataproc cluster, submits a PySpark job that reads the BigQuery raw table, compares with staging using SHA256 hash, detects delta records, and loads them into the staging table as a new partition. The cluster is immediately destroyed after the job completes. Dataplex sits as a unified governance layer across all three BQ layers — organising tables into Lakes and Zones, auto-discovering metadata, running automated DQ scans after every partition write, tracking end-to-end data lineage, and enforcing zone-level IAM policies. A live Looker Studio dashboard reads from BQ audit tables (including Dataplex DQ results) and provides full pipeline observability.

Data Flow Sequence
1 Composer triggers DAG on last day of month (02:00 UTC)
2 DataprocCreateClusterOperator spins up ephemeral cluster
3 PySpark job reads Raw → computes SHA256 hashes → Left Anti Join
4 Delta records (NEW + CHANGED) written to Staging partition (APPEND)
5 Spark SQL transformation → write to Curation partition (OVERWRITE)
6 Dataplex DQ scans triggered on both staging + curation writes
7 DataprocDeleteClusterOperator (trigger_rule=all_done) — always runs
8 Audit log written → Looker Studio dashboard auto-refreshes

✅ Pros & ❌ Cons

✅ PROZero Idle Compute CostDataproc cluster exists only for job duration (~60–90 min/month)
✅ PROFull OrchestrationComposer manages scheduling, retries, cluster lifecycle, DQ scans, validation in a single DAG
✅ PROEphemeral = Clean StateEach run gets a fresh cluster — no state pollution, no version drift
✅ PROAutomated Data QualityDataplex DQ scans run after every staging and curation write automatically
✅ PROUnified Data GovernanceDataplex Lake + Zones enforce consistent IAM, discovery, and access control
✅ PROAutomatic Data LineageRaw → Staging → Curation lineage tracked automatically via Dataplex + BQ lineage API
✅ PROMetadata CatalogAll BQ tables auto-registered in Dataplex Catalog with business tags
✅ PROZone-Level IAMAccess policies set at zone level — not per table — simplifies governance at scale
✅ PROLive Observability Dashboard5-page Looker Studio report including Dataplex DQ results page
✅ PROMonth-End PatternIdeal for regulatory, financial, and compliance pipelines
❌ CONCluster Startup Latency3–6 min Dataproc spin-up — unsuitable for sub-hour or latency-sensitive workloads
❌ CONComposer Always-On CostComposer 2 environment ~$300–800/month regardless of DAG frequency
❌ CONDataplex DQ Scan LatencyEach DQ scan adds 5–15 min to total pipeline duration
❌ CONDataplex Setup OverheadInitial Lake, Zone, Asset, and DQ rule configuration requires upfront planning
❌ CONMulti-Technology ExpertiseRequires knowledge across PySpark, Dataproc, Dataplex DQ, and BQ connector tuning
