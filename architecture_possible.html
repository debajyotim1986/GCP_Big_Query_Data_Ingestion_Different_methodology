01

🗄️ Pure BigQuery SQL + Scheduled Queries

Zero-infrastructure — all logic in SQL, scheduled natively in BQ
Architecture Wireframe
flowchart TD A([🕐 Cloud Scheduler\nor Scheduled Query]) --> B subgraph BQ["☁️ BigQuery Engine"] B[["📋 Step 1\nFull Snapshot Scan\nWatermark Filter"]] B --> C[["🔢 Step 2\nMD5 Hash Generation\nSpecific Cols or All Cols"]] C --> D[["🔍 Step 3\nDelta Detection\nLEFT JOIN vs Target"]] D --> E[["✅ Step 4\nDQ Rules\nCTE-based SQL Checks"]] E --> F{DQ\nPassed?} F -- Yes --> G[["💾 Step 5\nMERGE to Target Table"]] F -- No --> H[["⚠️ Step 5b\nWrite to Error Table"]] G --> I[["🔒 Step 6\nUpdate Process Control\nAdvance Watermark"]] I --> J[["📝 Step 7\nInsert to Report Table"]] H --> J end subgraph TABLES["📂 BQ Tables"] SRC[(Source Table)] TGT[(Target Table)] ERR[(Error Table)] CTL[(Process Control)] RPT[(Report Table)] end B -.->|Read| SRC G -.->|Write| TGT H -.->|Write| ERR I -.->|Update| CTL J -.->|Insert| RPT style BQ fill:#1a237e,color:#fff,stroke:#3949ab style TABLES fill:#1b5e20,color:#fff,stroke:#388e3c style F fill:#f57f17,color:#000,stroke:#f9a825

All logic lives inside a single BigQuery query chain (CTEs or chained CREATE TEMP TABLE statements). The pipeline runs entirely within BQ — no Python, no clusters. Cloud Scheduler or BQ Scheduled Queries triggers execution. BQ does snapshot scan, hash generation, delta detection, DQ checks, MERGE, and writes all audit records in sequence.

✅ Pros

  • Zero infrastructure — no clusters, containers, or environments
  • Fastest time to production — days, not weeks
  • Native BQ performance — petabyte-scale columnar speed
  • Pay-per-query — no always-on cost
  • Full SQL auditability — every step in BQ Job History
  • Native scheduling — BQ Scheduled Queries has built-in retry

❌ Cons

  • DQ logic is SQL-only — no ML-based or cross-system checks
  • Dynamic SQL via EXECUTE IMMEDIATE gets messy at scale
  • No unit testing framework for SQL procedures
  • Limited error branching — IF/CASE chains replace proper exceptions
  • Hard to version-control — SP bodies not easily diffable
  • No row-level quarantine routing natively
📌 Best For: Source is BigQuery, DQ rules are SQL-expressible, team is SQL-first, fast delivery needed.
02

🌊 Cloud Dataflow (Apache Beam)

Managed distributed processing — best for non-BQ sources and complex DQ
Architecture Wireframe
flowchart TD TRIGGER([🕐 Cloud Scheduler\nor Composer]) --> JOB subgraph JOB["🌊 Dataflow Flex Template Job"] R[["📥 Source Reader\nBoundedSource\nBQ / GCS / Cloud SQL"]] R --> SNAP[["📸 Snapshot Transform\nApply watermark filter"]] SNAP --> SI[["🔑 Side Input Load\nTarget PKs + Hashes"]] SI --> DELTA[["🔍 Delta Detection\nCoGroupByKey\nHash comparison"]] DELTA --> DQ[["✅ DQ Transform\nCustom DoFn\nPer-row validation"]] DQ --> SPLIT{Route\nOutput} SPLIT --"PASS"--> MERGE[["💾 Target Writer\nBQ Storage Write API"]] SPLIT --"QUARANTINE"--> Q[["🚫 Dead Letter Sink\nBQ Error Table"]] SPLIT --"WARN"--> W[["📋 Warning Logger\nBQ Audit Table"]] MERGE --> AUDIT[["📊 Audit Sink\nRun Summary"]] Q --> AUDIT W --> AUDIT end subgraph BQ_OUT["📂 BigQuery Sinks"] TGT2[(Target Table)] DL[(Dead Letter)] AUD[(Audit Table)] CTL2[(Process Control)] end MERGE -.->|Write| TGT2 Q -.->|Write| DL AUDIT -.->|Write| AUD AUDIT -.->|Update| CTL2 style JOB fill:#0d47a1,color:#fff,stroke:#1565c0 style BQ_OUT fill:#1b5e20,color:#fff,stroke:#388e3c style SPLIT fill:#e65100,color:#fff,stroke:#f57c00

A Flex Template (containerized Apache Beam job) triggered per schedule. Reads full snapshot using a BoundedSource, loads target hashes as a side input, routes records through a DQ DoFn using TaggedOutputs, then writes to appropriate sinks. All transforms are distributed across Dataflow workers with auto-scaling.

✅ Pros

  • Best for non-BQ sources — GCS, Cloud SQL, Pub/Sub, Spanner
  • True distributed processing — auto-scales to any volume
  • Rich DQ — custom Python/Java DoFn can call ML models or APIs
  • Native dead letter routing — TaggedOutputs split PASS/FAIL/WARN
  • Exactly-once delivery guarantees
  • Flex Templates are portable across projects

❌ Cons

  • Higher complexity — Beam programming model has a learning curve
  • Startup latency — job startup takes 2–5 minutes
  • Cost overhead — per-worker-hour even for short jobs
  • Overkill for BQ-to-BQ — native BQ SQL is faster and cheaper
  • Side input memory limits for large target hash sets
  • Template images need building and pushing to Artifact Registry
📌 Best For: Source is GCS/Cloud SQL/external API, dataset is TB+, DQ requires custom Python, multiple output sinks needed.
03

🎼 Cloud Composer + BigQuery Operators

Best orchestration — multi-step DAG with DQ gate, retries, SLA monitoring
Architecture Wireframe
flowchart TD SCHED([🕐 Cloud Scheduler\nDAG Trigger]) --> DAG subgraph DAG["🎼 Cloud Composer DAG"] T1["⚙️ Task 1\nload_config\nPythonOperator"] T1 --> T2["📸 Task 2\nextract_snapshot\nBigQueryInsertJobOperator"] T2 --> T3["🔍 Task 3\ndetect_delta\nBigQueryInsertJobOperator"] T3 --> T4["✅ Task 4\nrun_dq_checks\nBigQueryCheckOperator"] T4 --> T5{BranchPython\nOperator\nDQ Gate} T5 --"PASS"--> T6["💾 Task 5a\nmerge_to_target\nBigQueryInsertJobOperator"] T5 --"FAIL"--> T7["🚫 Task 5b\nwrite_dq_errors"] T6 --> T8["🔒 Task 6\nupdate_watermark\nPythonOperator"] T7 --> T8 T8 --> T9["📊 Task 7\nwrite_run_report"] T9 --> T10["📣 Task 8\nsend_alert\nCloud Monitoring"] end style DAG fill:#1a237e,color:#fff,stroke:#283593 style T5 fill:#e65100,color:#fff,stroke:#bf360c

Each pipeline step is a separate Airflow task connected by dependencies. BranchPythonOperator acts as a DQ gate — halting or routing based on DQ results. BigQuery does all heavy SQL work; Composer provides orchestration, retry logic, SLA monitoring, and enterprise alerting. XCom passes run summaries between tasks.

✅ Pros

  • Best orchestration — full DAG control, retries, SLAs, branching
  • Native BQ integration — BigQueryOperator family
  • Great Expectations support via GreatExpectationsOperator
  • Visual DAG monitoring — Airflow UI with task-level status
  • XCom for inter-task data — run summaries passed cleanly
  • Enterprise alerting — PagerDuty, Slack, email built-in

❌ Cons

  • Composer environment cost — always-on ~$300–800/month
  • Parse-time overhead — complex DAGs slow Composer scheduler
  • Airflow complexity — steep learning curve for DAG patterns
  • Task overhead — each task has 30–60s startup cost
  • Not for real-time — batch scheduler only
  • Worker resource limits for PythonOperator tasks
📌 Best For: Multi-step pipeline with complex dependencies, enterprise monitoring needed, multiple teams sharing one orchestration platform.

⚡ Dataproc (PySpark / Serverless)

Distributed Spark processing — best for GCS-sourced large-scale data
Architecture Wireframe
flowchart TD TRIG([🕐 Composer Trigger\nor REST API]) --> SUBMIT SUBMIT["📤 DataprocSubmitJobOperator"] --> DP subgraph DP["⚡ Dataproc Serverless / Cluster"] READ[["📥 PySpark Reader\nspark.read.bigquery()\nor spark.read.parquet(gcs)"]] READ --> FILTER[["🔎 Watermark Filter\ndf.filter(col > last_wm)"]] FILTER --> HASH[["🔢 Hash Generation\nwithColumn MD5 concat"]] HASH --> JOIN[["🔍 Delta Detection\ndf.join on PKs\nhash mismatch filter"]] JOIN --> DQ[["✅ DQ Checks\nPySpark validators\nor PyDeequ"]] DQ --> ROUTE{Split\nDataFrame} ROUTE --"Pass"--> WRITE[["💾 BQ Writer\nspark-bigquery-connector"]] ROUTE --"Fail"--> ERRWRITE[["⚠️ Error Writer\nerror_df to BQ"]] WRITE --> WM[["🔒 Watermark Update\nbq_client call"]] ERRWRITE --> WM WM --> AUD[["📊 Audit Record\nrun summary insert"]] end subgraph STORE["📂 Storage"] GCS2[GCS Parquet/CSV/Avro] BQ2[BigQuery Source/Target] end READ -.->|Read| GCS2 READ -.->|Read| BQ2 WRITE -.->|Write| BQ2 style DP fill:#0d47a1,color:#fff,stroke:#1565c0 style STORE fill:#1b5e20,color:#fff,stroke:#388e3c style ROUTE fill:#e65100,color:#fff,stroke:#bf360c

PySpark job runs on Dataproc Serverless (recommended) or a standard cluster. Source data from GCS or BQ is read into DataFrames. Delta detection uses a DataFrame join. DQ is applied as column-level filters or PyDeequ rules. Clean delta is written to BQ via the spark-bigquery-connector. Triggered by Composer or direct REST API call.

✅ Pros

  • Best for GCS-sourced large data — native Parquet/ORC/Avro reader
  • Full Spark ecosystem — PyDeequ, Spark ML, custom UDFs
  • Serverless Dataproc — no cluster management, pay-per-job
  • Rich DataFrame API — complex delta logic in clean Python
  • Multi-format support — Hive, Delta Lake, Iceberg natively
  • Familiar for data engineers — Spark is industry standard

❌ Cons

  • Startup time — even Serverless Spark takes 2–4 minutes
  • Expensive for small jobs — minimum billing unit
  • Cluster management for standard clusters — sizing, patching
  • BQ write bottleneck — spark-bigquery-connector slower than native
  • Memory-bound joins — large hash side loads can exceed worker memory
  • Overengineered for BQ-to-BQ pipelines
📌 Best For: Source data is on GCS (Parquet/CSV/Avro), very large dataset, Spark expertise available, Hive/Iceberg ecosystem integration needed.

📦 Cloud Run Jobs

Containerised Python batch — low-cost, fast iteration, medium-scale
Architecture Wireframe
flowchart TD SCHED2([🕐 Cloud Scheduler\nHTTP trigger]) --> CRJ subgraph CRJ["📦 Cloud Run Job — Containerised Python"] INIT[["🚀 Init\nLoad config from BQ\nGet watermark"]] INIT --> LOCK[["🔒 Acquire Lock\nAtomic BQ MERGE"]] LOCK --> SNAP2[["📸 Snapshot Query\nBQ client watermark filter"]] SNAP2 --> DELTA2[["🔍 Delta Detection\nPolars or BQ query join"]] DELTA2 --> DQ2[["✅ DQ Validation\nPandera / Great Expectations"]] DQ2 --> BRANCH{DQ Result} BRANCH --"REJECT"--> ERRLOG[["⚠️ Error Log\nWrite counts to BQ"]] BRANCH --"Pass"--> WRITE2[["💾 BQ Write\nStorage Write API\nMERGE mode"]] ERRLOG --> PC[["🔒 Update Process Control"]] WRITE2 --> PC PC --> RPT2[["📊 Audit Record\nInsert report table"]] RPT2 --> CLEAN[["🧹 Cleanup\nRelease lock"]] end subgraph INFRA2["🏗️ Infrastructure"] AR2[Artifact Registry\nDocker Image] SEC[Secret Manager\nCredentials] end subgraph BQ3["📂 BigQuery"] SRC4[(Source)] TGT4[(Target)] ERR4[(Error Table)] CTL4[(Process Control)] end CRJ --- INFRA2 SNAP2 -.->|Query| SRC4 WRITE2 -.->|Write| TGT4 ERRLOG -.->|Insert| ERR4 PC -.->|Update| CTL4 style CRJ fill:#004d40,color:#fff,stroke:#00695c style INFRA2 fill:#37474f,color:#fff,stroke:#546e7a style BQ3 fill:#1b5e20,color:#fff,stroke:#388e3c style BRANCH fill:#e65100,color:#fff,stroke:#bf360c

A containerised Python application runs as a Cloud Run Job triggered by Cloud Scheduler. Uses the BQ Python client for queries and writes. DQ implemented via Pandera, Great Expectations, or custom rules in-memory. BQ Storage Write API handles efficient bulk writes. All state persisted to BQ between runs — fully stateless container.

✅ Pros

  • Low cost — pay only for execution time, no always-on infra
  • Simple deployment — one Docker image, one Scheduler job
  • Full Python flexibility — any library for DQ, logging, transforms
  • Fast cold start — seconds vs minutes for Dataflow/Dataproc
  • Easy local development — run container locally for E2E testing
  • Secret Manager integration — credentials cleanly injected

❌ Cons

  • Memory-bound — 32GB max; not for TB+ datasets
  • No distributed processing — single-container only
  • No built-in retry DAG — all-or-nothing retries
  • No visual pipeline view — no Airflow-style UI
  • Custom metrics needed — must push to Cloud Monitoring manually
  • Stateless per run — all state must persist to BQ/GCS
📌 Best For: Medium-scale BQ-to-BQ, Python-first team, low-cost requirement, fast iteration cycle.

🏛️ BQ + Dataplex DQ (Enterprise Governed)

Full governance — lineage, DQ catalog, regulatory compliance built-in
Architecture Wireframe
flowchart TD SCHED3([🕐 Cloud Composer\nDAG Trigger]) --> T_SNAP T_SNAP["⚙️ Task 1\nextract_snapshot\nBigQueryInsertJobOperator"] T_SNAP --> T_DELTA["⚙️ Task 2\ndetect_delta\nBigQueryInsertJobOperator"] T_DELTA --> T_DQ["⚙️ Task 3\nTrigger Dataplex DQ Scan\nDataplexDataQualityJobTrigger"] subgraph DATAPLEX["🔷 Cloud Dataplex DQ Engine"] DQ_YAML["📋 DQ Rules YAML\nNull / Range / Regex\nUniqueness / Custom SQL"] DQ_SCAN["🔍 Managed DQ Scan\nServerless execution"] DQ_RESULT["📊 DQ Results\nRow-level pass/fail scores"] DQ_YAML --> DQ_SCAN --> DQ_RESULT end T_DQ --> DATAPLEX DATAPLEX --> T_CHECK["⚙️ Task 4\nPoll DQ Status\nPythonOperator"] T_CHECK --> GATE{DQ Gate} GATE --"PASS"--> T_MERGE["⚙️ Task 5a\nmerge_to_target\nBigQueryInsertJobOperator"] GATE --"FAIL"--> T_HALT["⚙️ Task 5b\nhalt_and_alert\nCloud Monitoring"] T_MERGE --> T_WM["⚙️ Task 6\nadvance_watermark"] T_WM --> T_RPT["⚙️ Task 7\nwrite_report"] subgraph CATALOG["🗂️ Dataplex Catalog"] LINEAGE[Column-level Lineage] DQ_HIST[DQ Score History] end DATAPLEX -.->|Publish| LINEAGE DATAPLEX -.->|Update| DQ_HIST style DATAPLEX fill:#311b92,color:#fff,stroke:#4527a0 style CATALOG fill:#1b5e20,color:#fff,stroke:#2e7d32 style GATE fill:#e65100,color:#fff,stroke:#bf360c

Composer orchestrates staging and MERGE steps. Dataplex Data Quality acts as the managed DQ engine — YAML-configured rules run serverlessly against the delta staging table. Results are stored in BQ and surfaced in Dataplex Catalog with lineage tracking and historical DQ scores. Fully governed and auditable without writing DQ code.

✅ Pros

  • Enterprise governance built-in — lineage, catalog, DQ scores
  • Zero DQ code — rules defined in YAML, no implementation needed
  • Historical DQ trending — rule pass/fail scores over time
  • Column-level lineage tracking from source to target
  • Regulatory compliance ready — GDPR/CCPA audit trail
  • Managed scaling — Dataplex DQ scans are fully serverless

❌ Cons

  • Asynchronous DQ — Composer must poll; adds 5–15 min latency
  • Dataplex cost — per-scan pricing for frequent small runs
  • Limited custom DQ — complex cross-system checks require workarounds
  • Setup complexity — Dataplex zone/lake/catalog configuration
  • Scan coverage — Dataplex samples by default; full scan needs config
  • GCP lock-in — Dataplex DQ rules are not portable
📌 Best For: Enterprise data platform with compliance requirements, data catalog, lineage tracking, regulatory audit trail, and centralised DQ governance.

🔀 Recommended Hybrid Architecture

Composer orchestration + Python-generated BQ SQL + config-driven DQ framework
Full Enterprise Hybrid Wireframe
flowchart TD SCHED4([🕐 Cloud Scheduler\n2:00 AM UTC Daily]) --> COMP subgraph COMP["🎼 Cloud Composer — Orchestration Layer"] D1["discover_tables\nPythonOperator\nLoad table_registry from BQ"] D1 --> PARA subgraph PARA["⚡ Parallel Tasks — max 5 concurrent"] D2["extract_snapshot\nBigQueryInsertJobOperator\nWatermark-filtered scan"] D3["detect_delta\nBigQueryInsertJobOperator\nDynamic hash SQL"] D4["run_dq_rules\nPythonOperator\nConfig-driven BQ DQ"] D5{DQ Gate\nBranchOperator} D6["merge_to_target\nBigQueryInsertJobOperator"] D7["quarantine_failed\nBigQueryInsertJobOperator"] D2 --> D3 --> D4 --> D5 D5 --PASS--> D6 D5 --QUARANTINE--> D7 D7 --> D6 end D6 --> D8["advance_watermark\nPythonOperator"] D8 --> D9["write_audit_record\nBigQueryInsertJobOperator"] end D9 --> REPORT["write_dag_report\nTriggerRule.ALL_DONE"] subgraph BQ_FW["📂 BQ framework_config Dataset"] R1[(table_registry)] R2[(dq_rules)] R3[(process_control)] R4[(run_audit)] R5[(dq_error_log)] R6[(quarantine_records)] end subgraph ALERT["🔔 Alerting"] CM2[Cloud Monitoring] SL2[Slack / PagerDuty] end D1 -.->|Read| R1 D4 -.->|Read rules| R2 D8 -.->|Update| R3 D9 -.->|Insert| R4 D4 -.->|Write errors| R5 D7 -.->|Write rows| R6 REPORT -.-> CM2 CM2 -.-> SL2 style COMP fill:#1a237e,color:#fff,stroke:#3949ab style PARA fill:#0d47a1,color:#fff,stroke:#1565c0 style BQ_FW fill:#1b5e20,color:#fff,stroke:#388e3c style ALERT fill:#4a148c,color:#fff,stroke:#6a1b9a style D5 fill:#e65100,color:#fff,stroke:#bf360c

The recommended pattern for most enterprise BQ-to-BQ pipelines. Composer dynamically discovers all active tables from table_registry and generates one TaskGroup per table. Python generates dynamic BQ SQL at runtime — no hardcoded column lists. DQ rules are loaded from dq_rules config table. Watermark is only advanced on success. All outcomes written to audit tables regardless of pass/fail.
